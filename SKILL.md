# Codex Daily Report Skill

这个 skill 用来把本机 Codex 会话记录整理成“当天工作日报”。

核心目标只有一个：根据某一天、某个工作区里的真实对话，还原当天做了什么，再写成非技术同事也能看懂的日报。

默认数据源：

- `~/.codex/state_5.sqlite`
- `~/.codex/sessions/.../*.jsonl`

## 先确认的输入

优先确认下面四件事：

- 目标日期
- 目标工作区
- 总工时
- 输出位置

默认规则：

- 用户没指定工作区时，优先使用当前工作区
- 用户明确给了 `8h`、`10h` 之类总工时，就按这个总数拆分
- 用户没说是否需要“明日计划”，不要强行补

## 输出路径优先级

不要把输出目录写死到某个项目。

按下面顺序处理：

1. 用户明确给了文件路径或目录，就用用户指定的位置。
2. 用户没给时，先在当前工作区查是否已有稳定的日报目录约定，再沿用原有目录。
3. 如果当前工作区没有明显约定，默认写到当前工作区下的 `./daily/{date}.md`。

如果目标文件已经存在：

- 默认先读取已有内容，再做补充或调整
- 除非用户明确要求覆盖，否则不要直接整份重写

## 快速入口

- 线程查询、`rollout_path` 查询、`jq` 抽消息命令：读 [`references/commands.md`](references/commands.md)

## 默认工作流

1. 确认日期、工作区、总工时、输出位置。
2. 用 `state_5.sqlite` 查这一天、这个工作区下的线程。
3. 取出每个线程对应的 `rollout_path`。
4. 用 `jq` 只抽目标日期的 `user_message`。
5. 判断哪些线程属于真实工作，哪些应该排除。
6. 按工作主线归并成 3 到 6 条工作事项。
7. 必要时再用当天代码、文件或提交情况做辅助校验，但不能喧宾夺主。
8. 写成自然、简短、可读的中文日报。

## 线程筛选原则

默认排除：

- 当前这条“让我写日报”的线程
- 只有测试语句、空消息、随手试命令的线程
- 纯闲聊、纯账号切换、纯工具自检且和当天交付无关的线程
- 用户明确说“不要写进去”的内容

默认保留：

- 需求沟通、方案梳理、页面或流程模拟、业务理解
- bug 排查、修复、联调、验证、上线准备
- 和实际交付有关的环境搭建、配置接入、测试环境排障
- 用户后续补充说明“这个也算今天工作”的线程

## 归并规则

优先按“真实工作主线”归并，不要按“开了几个会话”归并。

优先使用这些粒度：

- 一个需求理解或方案输出
- 一个独立 bug 的排查或修复
- 一条完整的联调、验证或接入工作
- 一个明确成体系的业务调研或文档梳理

不要这样写：

- 按上午、下午拆
- 按线程数逐条罗列
- 把一句话能说清的同类小动作硬拆成很多点

## 写作要求

- 面向非技术读者，优先写人话
- 一条事项里尽量写清“做了什么”和“产出了什么”
- 不要堆 SQL、接口名、类名、配置项，除非不写就解释不清
- 少写“同步、跟进、处理相关问题”这种空话，尽量写具体动作
- 如果当天有多个 bug，要按不同问题拆开，不要混成一句

## 工时拆分

- 总工时以用户明确要求为准
- 一般拆成 3 到 6 条事项
- 工时分配要和事项重量匹配，不要平均分
- 如果用户只要内容、不强调工时，可以先按事项写清楚，再决定是否补小时数

## 推荐格式

```md
# 工作内容

- **事项 A**：一句话写清今天完成的动作和结果。3h
- **事项 B**：一句话写清推进内容和当前结论。2h
- **事项 C**：一句话写清排查、联调或验证结果。3h

# 明日计划

- 下一步 1
- 下一步 2
```

如果用户只要“今日工作”，那就只输出 `# 工作内容`，不要额外加无关部分。

## 关键边界

- 这套流程的主依据是 Codex 会话记录，不是 Git。
- Git、文件改动、已有日报文件只用于辅助核对，不要反过来靠 Git 臆测整天的工作。
- 如果会话内容和代码状态有出入，优先根据用户当天的真实对话判断，再标记必要的推断。
- 如果当前工作区和输出目录都不明确，先优先使用当前工作区；仍然不清楚时，再让用户补充。

## 何时读取参考文件

- 需要精确 SQL 或 `jq` 命令时，读取 `references/commands.md`
- 需要查线程、查 `rollout_path`、抽用户消息时，优先复用参考命令，不要临时手写一套
